# 설계 결정 기록 (ADR)

프로젝트의 주요 결정과 그 이유. 결정을 뒤집으려면 여기에 사유를 추가로 기록한다.

## D1. 규칙 기반 + LLM 계층 구조 (LLM 전면 사용 기각)
- 결정: 일상 행동은 규칙/확률, 전환점 결정만 LLM
- 이유: 시민 1,000명 × 매 결정 LLM 호출 시 시뮬레이션 1회에 수십만 원.
  계층화 + 캐싱 + 배칭으로 회당 수백~수천 원까지 절감 가능
- 함의: Phase 2(규칙 코어) 완성 전 LLM 통합 금지

## D2. 집단 수준 재현을 목표로, 개인 정확성은 주장하지 않음
- 결정: "개별 에이전트가 실제 사람처럼 생각한다"는 주장 금지
- 이유: 증명 불가능하며 심사·영업에서 공격받는 지점.
  주장은 "집단 지표가 실제 통계와 ±X% 일치" + validation/ 백테스트 수치로만
- 함의: 발표 Q&A 1순위 예상 질문 ("AI 판단을 어떻게 믿나")의 공식 답변

## D3. 데모 지역 = 서울 성수동
- 결정: 성수동 1개 동으로 한정
- 이유: 서울만 생활인구·상권 추정매출이 공개되어 검증 가능.
  카페 밀도와 스토리성도 좋음. 경기권은 데이터 빈약
- 함의: 지역은 config로 관리. 타 지역 확장은 로드맵 항목

## D4. 상품 = 리포트 단건 판매 (구독 기각)
- 결정: 대시보드 구독이 아닌 리포트 1건 단위 판매
- 이유: 창업 결정은 일회성 이벤트라 구독 동기가 없음.
  대시보드는 신뢰 형성 장치이지 상품이 아님

## D5. 엔진과 서비스 분리 (오픈소스 전략)
- 결정: engine/은 단독 실행 가능한 Python 패키지로 유지
- 이유: 오픈소스 개발자대회는 "다른 개발자가 가져다 쓰는 도구"가 강세.
  엔진=라이브러리 제출, 웹=공식 데모라는 이중 구조
- 함의: engine이 web/api에 의존하는 코드 금지

## D6. 검증 체계 우선, 정밀도는 장기 과제
- 결정: 이번 범위는 백테스트 파이프라인 구축까지. ±15%는 목표 아님
- 이유: 3.5주 내 정밀 캘리브레이션은 비현실적.
  "검증 방법론의 존재" 자체가 심사 점수. 정밀도는 가을 캡스톤 기간에

## D7. 업종 2등급제
- 결정: 검증 가능 업종(카페 등)만 정식 지원, 특이 업종은 "참고용" 명시
- 이유: 공공 매출 통계가 없는 업종은 백테스트 불가 → 오차 주장 불가.
  구분을 흐리면 신뢰 상품이라는 정체성이 무너짐

## D8. DB만 도커, 코드는 로컬
- 결정: docker-compose는 PostGIS 컨테이너만. Python/React는 각자 로컬
- 이유: PostGIS 설치는 OS별로 까탈스러워 환경 통일 필요.
  코드까지 도커화는 3인 팀에 오버엔지니어링

## D9. 소비 모델의 매장 선택은 현재 무작위, 향후 정교화 예정
- 결정: 현재는 "가까운 매장 5곳 중 무작위 1곳"으로 방문 매장을 선택
- 이유: 매장별 실제 인기도(리뷰 수, 방문자 수 등)를 판단할 근거 데이터가 아직 없음.
  임의로 가중치를 매기면 데이터가 아닌 추측이 되어 오히려 위험
- 향후 정교화 방향:
  - 매장 자체의 입지 인기도는 현재 로직으로도 자연 반영됨
    (동선이 겹치는 목 좋은 자리일수록 "가까운 5곳" 후보에 자주 포함되어 방문 카운터가 자연히 늘어남)
  - 개인 취향·단골 패턴(같은 시민이 이전 방문지를 다시 찾을 확률)은 아직 미반영, 향후 과제
  - 실제 인기 요인(브랜드력, 가격대 등)은 Phase 4 백테스트에서
    시뮬레이션 예측과 실제 매출의 괴리를 통해 역발견 후 반영 예정

## D10. LLM 제공사로 Gemini 선택
- 결정: 전환점 의사결정(D1)에 사용할 LLM 제공사로 Google Gemini(소형 모델)를 선택
- 이유:
  1) 절대금액 기준 비용이 제공사 중 가장 저렴한 축에 속함
  2) 이 작업(짧은 구조화 출력 반복 — 전환 여부·확률·이유)은 소형 모델로 충분한 난이도
  3) 클라이언트를 교체 가능한 구조(LLMClient 인터페이스)로 설계해 제공사 종속 위험을 낮춤
- 함의: Phase 4 백테스트에서 판단 품질이 기준에 못 미치면 재검토한다.
  구현체(engine/decision/gemini_client.py) 하나만 교체하면 되도록 설계했으므로
  다른 제공사로 바꾸는 전환 비용은 낮다
- 추가 발견(2026-07-21, what-if 실제 검증 중): 무료 티어의 `gemini-2.5-flash-lite`는
  **일일 요청 한도가 20회**로 매우 작다(`GenerateRequestsPerDayPerProjectPerModel-FreeTier`).
  분당 한도(RPM)뿐 아니라 일일 한도까지 있어, 데모·개발 중 반복 호출로 하루 치를
  금방 소진할 수 있음 — 실제 what-if 라이브 검증이 이 한도에 막힘. 유료 결제 설정
  또는 다음날 재시도로 해결 가능. 프로덕션 전환 시 유료 티어 전환이 사실상 필수
- 추가 발견 2(2026-07-23, 유료 결제 후 재검증 중): 일일 한도와 별개로 `gemini-2.5-flash-lite`가
  `503 UNAVAILABLE`("high demand")을 SDK 자체 재시도(tenacity)로도 못 넘길 만큼 자주 반환함.
  `engine/decision/gemini_client.py`에 지수 백오프 재시도(최대 4회, 5초→10→20→40초)를
  추가해 해결 — 실제로 이 재시도 덕분에 성수동 실좌표 what-if(7일치, 시민 1,000명)가
  3분 52초 만에 최종 성공(일평균 방문 353.4명, 전환 380명, 이탈 출처·시간대 분포 모두
  합리적으로 산출됨: 저녁 17~23시 집중, 21시 피크). 유료 결제로도 완전히 사라지는
  문제는 아니라는 뜻 — 재시도 로직은 계속 유지 필요
