# 설계 결정 기록 (ADR)

프로젝트의 주요 결정과 그 이유. 결정을 뒤집으려면 여기에 사유를 추가로 기록한다.

## D1. 규칙 기반 + LLM 계층 구조 (LLM 전면 사용 기각)
- 결정: 일상 행동은 규칙/확률, 전환점 결정만 LLM (Haiku급)
- 이유: 시민 1,000명 × 매 결정 LLM 호출 시 시뮬레이션 1회에 수십만 원.
  계층화 + 캐싱 + 배칭으로 회당 수백~수천 원까지 절감 가능
- 함의: Phase 2(규칙 코어) 완성 전 LLM 통합 금지

## D2. 집단 수준 재현을 목표로, 개인 정확성은 주장하지 않음
- 결정: "개별 에이전트가 실제 사람처럼 생각한다"는 주장 금지
- 이유: 증명 불가능하며 심사·영업에서 공격받는 지점.
  주장은 "집단 지표가 실제 통계와 ±X% 일치" + validation/ 백테스트 수치로만
- 함의: 발표 Q&A 1순위 예상 질문 ("AI 판단을 어떻게 믿나")의 공식 답변

## D3. 데모 지역 = 서울 성수동
- 결정: 성수동 1개 동으로 한정
- 이유: 서울만 생활인구·상권 추정매출이 공개되어 검증 가능.
  카페 밀도와 스토리성도 좋음. 경기권은 데이터 빈약
- 함의: 지역은 config로 관리. 타 지역 확장은 로드맵 항목

## D4. 상품 = 리포트 단건 판매 (구독 기각)
- 결정: 대시보드 구독이 아닌 리포트 1건 단위 판매
- 이유: 창업 결정은 일회성 이벤트라 구독 동기가 없음.
  대시보드는 신뢰 형성 장치이지 상품이 아님

## D5. 엔진과 서비스 분리 (오픈소스 전략)
- 결정: engine/은 단독 실행 가능한 Python 패키지로 유지
- 이유: 오픈소스 개발자대회는 "다른 개발자가 가져다 쓰는 도구"가 강세.
  엔진=라이브러리 제출, 웹=공식 데모라는 이중 구조
- 함의: engine이 web/api에 의존하는 코드 금지

## D6. 검증 체계 우선, 정밀도는 장기 과제
- 결정: 이번 범위는 백테스트 파이프라인 구축까지. ±15%는 목표 아님
- 이유: 3.5주 내 정밀 캘리브레이션은 비현실적.
  "검증 방법론의 존재" 자체가 심사 점수. 정밀도는 가을 캡스톤 기간에

## D7. 업종 2등급제
- 결정: 검증 가능 업종(카페 등)만 정식 지원, 특이 업종은 "참고용" 명시
- 이유: 공공 매출 통계가 없는 업종은 백테스트 불가 → 오차 주장 불가.
  구분을 흐리면 신뢰 상품이라는 정체성이 무너짐

## D8. DB만 도커, 코드는 로컬
- 결정: docker-compose는 PostGIS 컨테이너만. Python/React는 각자 로컬
- 이유: PostGIS 설치는 OS별로 까탈스러워 환경 통일 필요.
  코드까지 도커화는 3인 팀에 오버엔지니어링
